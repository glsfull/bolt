# Реализация

## 1. Что сделано на этапе 0

Этап 0 выполнен как инвентаризация текущего прототипа `project/` без изменения
исходного приложения. Зафиксированы эталонные экраны, карта навигации, основные
доменные явления, черновик модели данных и дизайн-токены, которые нужно
сохранить при последующей реализации.

Важно: папка `project/` остается только источником эталонов и требований. Для
реальной реализации нужно создать отдельные папки верхнего уровня `frontend/` и
`backend/`; доработка production-кода внутри `project/` не допускается.

## 3. Карта маршрутов и навигации

Текущий прототип не использует URL-роутер. Навигация реализована через состояние
`activeTab` в `project/src/pages/Dashboard.tsx`.

Решение для следующего этапа: карту маршрутов ниже считаем утвержденной как
целевую для React Native + Expo Router или web/mobile rewrite. URL должны стать
стабильными адресами экранов, а `activeTab` остается только описанием текущего
прототипа.

### 3.1. Публичная зона

- `/` в терминах будущего роутера: лендинг `Мои анализы`;
- якоря лендинга: `#features`, `#how`, `#pricing`, `#app`;
- модальные состояния: вход и регистрация через `AuthModal`.

### 3.2. Авторизованная зона

Предлагаемая будущая карта маршрутов:

| Текущий tab | Будущий маршрут | Экран |
| --- | --- | --- |
| `home` | `/app` | Обзор здоровья |
| `upload` | `/app/analysis/upload` | Загрузка анализа |
| `camera` | `/app/analysis/camera` | Сканер камеры |
| `results` | `/app/analysis/result/:id` | Расшифровка анализа |
| `history` | `/app/history` | История и динамика |
| `symptoms` | `/app/ai` | ИИ-диагностика |
| `meds` | `/app/medications` | Расписание лекарств |
| `risks` | `/app/risks` | Оценка рисков |
| `report` | `/app/report` | Отчет для врача |
| `encyclopedia` | `/app/encyclopedia` | Справочник |
| `profile` | `/app/profile` | Профиль здоровья |
| `programs` | `/app/programs` | Программы здоровья |
| `admin` | `/admin` | Панель администратора |

### 3.3. Навигационные группы

Боковое меню:

- `Главное`: обзор, загрузить анализ, сканер камеры, расшифровка;
- `Аналитика`: история, оценка рисков;
- `ИИ & Здоровье`: ИИ-диагностика, лекарства, программы здоровья;
- `Прочее`: отчет врачу, справочник, профиль;
- `Управление`: панель администратора только для роли admin.

Нижняя мобильная навигация:

- `Главная` -> `home`;
- `Анализы` -> `history`;
- `ИИ` -> `symptoms`;
- `Программы` -> `programs`;
- `Профиль` -> `profile`.

Решение по расхождению: в целевой нижней навигации оставляем пункт `Программы`,
как в текущем React-коде. Экран лекарств остается доступным из бокового меню,
профиля и быстрых переходов, но не занимает слот нижней навигации. Причина:
программы здоровья являются более широким долгосрочным сценарием и связывают
риски, анализы и рекомендации, а лекарства являются отдельной прикладной
функцией внутри группы `ИИ & Здоровье`.

## 5. Черновик модели данных

Черновик основан на `project/src/data/demoAnalysis.ts` и текущих экранах.

### 5.1. UserProfile

Поля:

- `fullName`;
- `age`;
- `sex`;
- `weightKg`;
- `heightCm`;
- `bloodType`;
- `chronicConditions`;
- `allergies`.

Связи:

- один профиль принадлежит одному пользователю;
- профиль используется для рисков, отчетов и программ здоровья.

### 5.2. Analysis

Поля:

- `id`;
- `date`;
- `type`;
- `typeLabel`;
- `fileName`;
- `markersCount`;
- `abnormalCount`;
- `healthScore`.

Связи:

- анализ принадлежит пользователю;
- анализ содержит набор `AnalysisMarker`;
- анализ может быть связан с исходным файлом в приватном хранилище.

### 5.3. AnalysisMarker

Поля:

- `id`;
- `name`;
- `value`;
- `unit`;
- `min`;
- `max`;
- `status`: `low`, `normal`, `high`, `critical`;
- `description`;
- `interpretation`;
- `recommendation`;
- `category`.

Связи:

- маркер принадлежит анализу;
- маркер участвует в трендах, рисках, ИИ-заключениях и программах здоровья.

Поля:

- `markerId`;
- `name`;
- `unit`;
- `min`;
- `max`;

Связи:

- тренд агрегируется по маркеру и пользователю;

### 5.5. RiskAssessment

Поля:

- `name`;

- `level`: `low`, `medium`, `high`;
- `color`;
- `icon`;
- `description`;
- `factors`;
- `recommendations`.

Связи:

- риск рассчитывается на основании профиля, истории и маркеров;
- риск может быть основанием для программы здоровья и отчета врачу.

### 5.6. MedicationSchedule

Дополнительные сущности для production:

Поля:

Связи:

Поля:

Связи:

### 5.9. EncyclopediaArticle

Поля:

- `id`;
- `title`;
- `category`;
- `summary`;
- `content`;

- `tags`.

Связи:

Поля:

- `provider`;
- `model`;
- `apiKey`;

- `limits`;

Важно: секреты AI/OCR не должны храниться или использоваться на клиенте.
Production-реализация должна переносить эти настройки в backend/admin-зону.

## 6. Дизайн-токены

Токены зафиксированы как черновой контракт для будущей реализации. Источники:
Tailwind-классы текущего React-кода и CSS из HTML-эталонов.

На следующем этапе токены нужно вынести из документа в отдельный модуль
`frontend/mobile/src/theme` будущей реализации. Документ остается источником требований, а код
должен импортировать значения из theme-модуля, чтобы экраны не дублировали
цвета, радиусы, размеры и тени вручную.

- Бейдж risk high: `red-100/200` + `red-800`.
- Инфоблок: `blue-50` + `blue-200` + `blue-800`.
- Карточка: белый фон, `border-gray-100`, `rounded-2xl` или
  `rounded-3xl`, `shadow-sm`.

### 6.8. Структура будущего theme-модуля

Минимальный контракт:

```text
frontend/mobile/src/theme/
  colors.ts
  typography.ts
  spacing.ts
  radius.ts
  shadows.ts
  components.ts
  index.ts
```

Содержимое:

- `colors.ts`: базовая палитра, статусы маркеров, риски, информационные блоки;
- `typography.ts`: семейства шрифтов, размеры, веса, line-height;
- `spacing.ts`: экранные отступы, gap между блоками, высоты topbar/bottom tabs;
- `radius.ts`: радиусы иконок, карточек, модальных окон, mockup-панелей;
- `shadows.ts`: тени карточек, floating-панелей и модальных окон;
- `components.ts`: готовые токены для кнопок, карточек, бейджей, инфоблоков,
  bottom tabs и marker cards.

Правило использования: новые экраны сначала берут значения из `theme`, а
локальные ad hoc значения допускаются только для единичной визуализации,
которую затем нужно либо добавить в `theme`, либо явно оставить как исключение.

## 7. Production-схема Supabase

В репозитории уже есть черновые миграции:

- `project/supabase/migrations/20260513172613_create_user_profiles.sql`;
- `project/supabase/migrations/20260513174321_create_analyses_medications_tables.sql`;
- `project/supabase/migrations/20260513180400_create_health_extended_tables.sql`;
- `project/supabase/migrations/20260513184155_add_admin_settings_and_health_programs.sql`.

Их нужно использовать как стартовую модель, но не переносить в production без
ревизии. Целевая схема должна нормализовать данные, добавить статусы обработки,
файловое хранилище, аудит и индексы.

### 7.1. Базовые таблицы

Минимальный production-набор:

- `profiles`: пользовательский профиль, связанный с `auth.users`;
- `health_profiles`: медицинские параметры пользователя;
- `analysis_files`: исходные PDF/фото, путь в private Storage, MIME type,
  размер, checksum, статус загрузки;
- `analyses`: событие анализа, тип, дата, статус обработки, итоговый health
  score, ссылка на `analysis_files`;
- `analysis_markers`: нормализованные показатели, значения, единицы, референсы,
  статус и категория;
- `marker_reference_ranges`: справочник референсов по полу, возрасту, единицам
  и лабораторному типу;
- `analysis_processing_jobs`: очередь обработки с этапами OCR, AI-normalization,
  validation, completed, failed;
- `symptom_sessions`: запросы ИИ-диагностики и структурированные ответы;
- `risk_assessments`: результаты расчета рисков;
- `medications`: лекарства и добавки;
- `medication_intakes`: факты приема;
- `medication_reminders`: настройки уведомлений;
- `health_programs`: шаблоны программ;
- `user_health_programs`: назначения программ пользователям и прогресс;
- `encyclopedia_articles`: справочник показателей;
- `doctor_reports`: сгенерированные отчеты и ссылки на PDF;
- `admin_settings`: несекретные настройки AI/OCR;
- `audit_log`: административные и чувствительные изменения.

### 7.2. Обязательные правила схемы

- Включить RLS для всех таблиц с пользовательскими и медицинскими данными.
- Все пользовательские строки должны иметь `user_id` и политики доступа по
  `auth.uid()`.
- Файлы анализов и PDF-отчеты хранить только в private buckets.
- Для доступа к файлам использовать signed URLs с коротким TTL.
- Секреты AI/OCR хранить в Supabase Edge Function secrets, а не в таблицах и не
  на клиенте.
- Для processing jobs хранить `status`, `attempts`, `error_code`,
  `error_message`, `started_at`, `finished_at`.
- Для маркеров хранить исходное OCR-значение и нормализованное значение, чтобы
  можно было проверять качество распознавания.
- Для медицинских рекомендаций хранить версию промпта, модель, провайдера и
  дату генерации.

## 8. Границы backend-функций

Все функции ниже должны выполняться на backend-стороне через Supabase Edge
Functions или отдельный сервис. Клиент не должен обращаться к OCR/AI-провайдерам
напрямую.

### 8.1. `upload-analysis-file`

Ответственность:

- проверить тип, размер и принадлежность файла пользователю;
- создать запись `analysis_files`;
- выдать signed upload URL или принять файл через защищенный endpoint;
- создать черновой `analysis` со статусом `uploaded`.

Не делает: OCR, AI-интерпретацию, медицинские выводы.

### 8.2. `run-ocr`

Ответственность:

- получить файл из private Storage;
- отправить файл OCR-провайдеру или локальному OCR-движку;
- сохранить сырой текст, координаты блоков при наличии и confidence;
- перевести job в статус `ocr_completed` или `failed`.

Не делает: диагностику, рекомендации, расчет рисков.

### 8.3. `interpret-analysis`

Ответственность:

- нормализовать OCR-текст в структурированные показатели;
- привести единицы измерения и референсы к внутренней модели;
- определить статусы `low`, `normal`, `high`, `critical`;
- сформировать объяснение и рекомендации по каждому маркеру;
- сохранить `analysis_markers` и итоговую сводку.

Не делает: хранение файлов и прямую работу с камерой.

### 8.4. `risk-assessment`

Ответственность:

- брать профиль, историю анализов и актуальные маркеры;
- рассчитывать риски по версионированным правилам;
- сохранять факторы, score, level и рекомендации;
- отдавать клиенту только структурированный результат.

Не делает: замену врачебной диагностики и назначение лечения.

### 8.5. `symptom-checker`

Ответственность:

- принять текст симптомов и контекст последних анализов;
- вызвать AI-модель через защищенный backend;
- вернуть возможные состояния, срочность и рекомендуемых специалистов;
- сохранить сессию и версию модели/промпта.

Не делает: постановку диагноза и экстренную медицинскую маршрутизацию без
явного disclaimer.

### 8.6. `generate-doctor-report`

Ответственность:

- собрать профиль, последние анализы, динамику, риски и лекарства;
- сформировать PDF на backend;
- сохранить PDF в private Storage;
- создать `doctor_reports` и короткоживущую ссылку для просмотра или скачивания.

Не делает: публичную бессрочную ссылку и хранение PDF в открытом bucket.

### 8.7. `admin-settings`

Ответственность:

- читать и изменять несекретные настройки AI/OCR;
- проверять роль `admin`;
- писать изменения в `audit_log`;
- управлять feature flags, лимитами и выбранными моделями.

Не делает: отдачу секретных ключей клиенту.

## 9. Медицинское правило отказа от ответственности

Дисклеймер должен быть обязательным UI-элементом на экранах:

- `ИИ-диагностика`;
- `Оценка рисков`;
- `Отчет для врача`.

Базовый текст:

> Информация носит справочный характер и не является диагнозом, назначением
> лечения или заменой консультации врача. При ухудшении самочувствия,
> выраженных симптомах или экстренной ситуации обратитесь за медицинской
> помощью.

Правила применения:

- на экране ИИ дисклеймер показывается до запуска анализа симптомов и рядом с
  готовым ответом;
- на экране рисков дисклеймер расположен рядом с информационным блоком о
  расчетном характере оценки;
- в отчете врача дисклеймер включается в предпросмотр и PDF;
- критические значения маркеров должны сопровождаться отдельной рекомендацией
  обратиться к врачу;
- текст нельзя скрывать настройками пользователя или тарифом.

## 10. Эталоны, расхождения и вопросы для утверждения

- Нижняя навигация утверждена в варианте текущего React-кода: `Главная`,
  `Анализы`, `ИИ`, `Программы`, `Профиль`.
- Экраны загрузки, сканера, отчета, справочника, профиля, программ здоровья и
  админ-панели есть в React-коде, но не имеют отдельных HTML-эталонов в
  `project/screenshot_*.html`. Для полной UI parity их нужно дополнительно
  зафиксировать скриншотами.
- Для авторизации нужен отдельный эталон модального окна входа и регистрации.
- Админ-панель сейчас доступна внутри клиентского dashboard для admin-email.
  В production ее нужно вынести в отдельную защищенную зону.
- Черновые Supabase migrations присутствуют в `project/supabase/migrations`, но
  требуют production-ревизии RLS, индексов, аудита, job statuses и Storage.

### 10.1. Экраны, которые нужно доснять

Обязательный список недостающих эталонов:

- загрузка анализа: пустое состояние, выбранный файл, обработка, ошибка;
- сканер: камера, выбор из галереи, обработка OCR, ошибка доступа к камере;
- отчет врача: предпросмотр, генерация PDF, готовый PDF/ссылка;
- справочник: список, поиск, карточка показателя;
- профиль: заполненное и пустое состояние;
- программы здоровья: список, карточка программы, активная программа;
- админ-панель: настройки AI/OCR, лимиты, выключенные функции;
- авторизация: вход, регистрация, ошибка, восстановление доступа.

Формат хранения: новые файлы должны продолжить существующий шаблон
`project/screenshot_XX_<screen>.html`, чтобы ревью могло сравнивать экраны
последовательно.

## 13. Что сделано на этапе 3

Этап 3 выполнен как переход от статического мобильного каркаса к первому
рабочему продуктному слою. Цель этапа - связать основные экраны с единым
read layer, перенести критичные сценарии анализа и закрепить backend-контракты,
не подменяя их клиентскими имитациями production-логики.

Результат:

- добавлен `frontend/mobile/src/services/supabase/readLayer.ts`, который читает
  профиль, историю анализов, программы, предупреждения и расшифровку анализа
  через Supabase при наличии env-настроек;
- сохранен fallback на `frontend/mobile/src/features/demoData.ts`, чтобы
  мобильное приложение оставалось запускаемым без подключенного Supabase;
- перенесен экран `analysis/result/[id]` с категориями показателей, карточками
  маркеров, статусами `low`, `normal`, `high`, `critical` и шкалой референсов;
- перенесены экраны `analysis/upload` и `analysis/camera` с клиентским
  контрактом обработки через `analysis_jobs` и статусами `uploaded`,
  `ocr_processing`, `ai_processing`, `completed`;
- перенесены экраны `risks` и `report`, включая обязательный медицинский
  disclaimer для оценки рисков и отчета врачу;
- контрактный тест расширен проверкой Supabase read layer, demo fallback,
  статусов маркеров, job-status contract и отсутствия generic placeholders на
  перенесенных маршрутах.

Ограничения этапа:

- Supabase read layer пока является тонким адаптером и не заменяет production
  миграции, RLS-политики и Edge Functions;
- загрузка, OCR, AI-нормализация и генерация PDF зафиксированы как контракты,
  но backend-исполнение еще не реализовано;
- Expo web screenshot review не завершен, потому что пакет
  `frontend/mobile` пока не содержит web-зависимости `react-native-web` и
  `react-dom`;
- экраны лекарств, справочника, детальной программы, активной программы,
  авторизации и admin-зоны еще требуют полноценного переноса вместо
