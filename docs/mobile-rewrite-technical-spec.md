# Техническое задание и план переработки проекта `project`

## 1. Назначение документа

Документ подготовлен на основе анализа папки `project`, перенесенной из Bolt.new. Папка `project` рассматривается как исходный прототип только для чтения: ее файлы не изменяются, а используются как эталон функциональности, структуры экранов, визуального стиля и пользовательских сценариев.

В новой реализации нельзя дорабатывать код внутри `project/`. Для production-проекта нужно создать отдельные папки верхнего уровня `frontend/` и `backend/`, чтобы исходный прототип оставался неизменяемым эталоном, а новая кодовая база развивалась независимо.

Цель переработки: заново реализовать продукт как полноценное мобильное приложение для Google Play и App Store с сохранением дизайна, блоков, таблиц, навигации и пользовательских сценариев в максимально точной копии текущего прототипа.

## 2. Анализ текущего проекта

### 2.1. Текущий технологический стек

- Frontend: React 18, TypeScript, Vite.
- Стили: Tailwind CSS.
- Иконки: `lucide-react`.
- Backend/Auth/DB: Supabase JS client, Supabase SQL migrations.
- Формат: web-приложение, адаптированное под мобильный экран, но не нативное мобильное приложение.
- Данные: значительная часть экранов работает на демо-данных из `src/data/demoAnalysis.ts`.

### 2.2. Основные экраны и функции

- Лендинг сервиса "Мои анализы": описание возможностей, тарифы, отзывы, демо-результаты.
- Авторизация через Supabase.
- Личный кабинет с боковым меню и нижней мобильной навигацией.
- Обзор здоровья с индексом здоровья, быстрыми действиями, последним анализом и предупреждениями.
- Загрузка анализа: PDF/JPG/PNG/HEIC, выбор типа анализа, имитация этапов обработки.
- Сканер камеры/OCR: фото или галерея, этапы распознавания, демо-результаты.
- Расшифровка показателей анализа с группировкой по категориям.
- История анализов и динамика показателей.
- ИИ-диагностика симптомов.
- Расписание лекарств.
- Оценка рисков заболеваний.
- Отчет для врача с предпросмотром и будущей выгрузкой PDF.
- Справочник показателей.
- Профиль здоровья: пол, возраст, рост, вес, ИМТ, группа крови, хронические заболевания, аллергии.
- Программы здоровья: питание, образ жизни, прогресс, модальные карточки программ.
- Админ-панель: настройки AI/OCR-провайдеров, моделей, лимитов, включение функций.

### 2.3. Текущая база данных Supabase

В папке `project/supabase/migrations` уже заложены таблицы для:

- пользовательских профилей;
- анализов и медицинских показателей;
- расписания лекарств;
- расширенных health-таблиц;
- административных настроек;
- программ здоровья.

Эти миграции следует использовать как основу доменной модели, но перед переносом нужно провести ревизию ограничений, индексов, RLS-политик, аудита изменений и хранения медицинских файлов.

### 2.4. Ключевые ограничения текущего прототипа

- Это web-прототип, а не production mobile app.
- Камера, OCR, AI, PDF и часть сохранения данных реализованы как имитация или клиентская заготовка.
- Нет полноценного backend-слоя для обработки медицинских документов, очередей, статусов задач и защищенного хранения секретов.
- Медицинские данные требуют повышенного уровня безопасности, контроля доступа, логирования и юридических дисклеймеров.
- Админ-панель присутствует в клиентском интерфейсе и должна быть вынесена в отдельный защищенный web-admin или строго изолированную роль.

## 3. Рекомендуемый современный подход к мобильной разработке

### 3.1. Основная рекомендация

Рекомендуемый стек для переработки: **React Native + Expo + TypeScript + Expo Router**.

Причины:

- единая кодовая база для iOS и Android;
- публикация в App Store и Google Play из одного проекта;
- современная экосистема для камеры, push-уведомлений, файлов, биометрии, deep links и OTA-обновлений;
- близость к текущему React/TypeScript-прототипу, что снижает риск переписывания логики;
- возможность сохранить визуальную структуру Tailwind-подобными инструментами;
- хорошая интеграция с Supabase и Edge Functions.

### 3.2. UI-стек

Рекомендуемые варианты:

- `NativeWind` для Tailwind-подобного описания стилей в React Native.
- `react-native-svg` для круговых индикаторов, графиков и кастомной визуализации.
- `lucide-react-native` как аналог текущих иконок.
- `react-native-reanimated` и `react-native-gesture-handler` для модальных окон, bottom sheets, свайпов и плавных состояний.
- `@shopify/flash-list` для производительных списков истории анализов, справочника и программ.

### 3.3. Backend-стек

Рекомендуемый backend:

- Supabase Postgres как основная база данных.
- Supabase Auth для email/password, OAuth и magic links.
- Supabase Storage для PDF, фото анализов, отчетов.
- Supabase Edge Functions для AI/OCR, генерации отчетов, защищенной бизнес-логики.
- Очередь задач для тяжелой обработки документов: Supabase Queue/pgmq или внешний managed queue при росте нагрузки.
- Отдельный web-admin на Next.js или React Admin для административной панели.

## 4. Целевая архитектура

Целевая структура репозитория для реализации:

```text
frontend/
  mobile/
    app/
    src/
      components/
      features/
      services/
      domain/
      theme/
      lib/
  web-admin/
    app/
    src/
backend/
  supabase/
    migrations/
    functions/
  services/
    ai/
    ocr/
    reports/
  queues/
project/
  ...
```

`project/` остается только источником требований и эталонных экранов. `frontend/`
содержит мобильное приложение и при необходимости web-admin, а `backend/`
содержит Supabase-схему, Edge Functions, интеграции AI/OCR, очереди обработки и
генерацию отчетов.

### 4.1. Клиентское мобильное приложение

Предлагаемая структура мобильного приложения внутри `frontend/mobile`:

```text
frontend/mobile/
  app/
    (auth)/
    (tabs)/
      index.tsx
      history.tsx
      ai.tsx
      programs.tsx
      profile.tsx
    analysis/
      upload.tsx
      camera.tsx
      result/[id].tsx
    report/
    risks/
    encyclopedia/
  src/
    components/
    features/
      auth/
      dashboard/
      analyses/
      camera-scanner/
      risk-assessment/
      medications/
      health-programs/
      doctor-report/
      profile/
    services/
      supabase/
      api/
      ai/
      ocr/
    domain/
      analysis/
      profile/
      medication/
      program/
    theme/
    lib/
```

### 4.2. Backend и интеграции

```text
backend/
  supabase/
    migrations/
    functions/
      process-analysis/
      run-ocr/
      generate-doctor-report/
      symptom-checker/
      risk-assessment/
      admin-settings/
  services/
    ai/
    ocr/
    reports/
  queues/
```

Основной поток обработки анализа:

1. Пользователь загружает PDF/фото или делает снимок камерой.
2. Файл сохраняется в Supabase Storage в приватный bucket.
3. В таблице `analyses` создается запись со статусом `uploaded`.
4. Edge Function запускает OCR и извлечение показателей.
5. AI-модель нормализует показатели, единицы измерения и референсы.
6. Backend сохраняет структурированные маркеры в `analysis_markers`.
7. Клиент получает обновления статуса через polling или realtime.
8. После завершения пользователь видит экран расшифровки.

### 4.3. Безопасность

Обязательные требования:

- включить RLS для всех таблиц с пользовательскими медицинскими данными;
- запретить клиенту прямой доступ к секретным AI/OCR-ключам;
- хранить файлы анализов только в приватных buckets;
- использовать signed URLs с коротким TTL;
- вести журнал административных изменений;
- разделить роли `user`, `doctor`, `admin`;
- добавить медицинский disclaimer на экранах AI-диагностики, рисков и отчетов;
- предусмотреть удаление аккаунта и пользовательских медицинских данных.

## 5. Требования к точному переносу дизайна

### 5.1. Визуальная идентичность

Необходимо сохранить:

- бренд "Мои анализы";
- белый и светло-серый фон интерфейса;
- emerald/teal акцент для основного действия;
- темные hero/summary-блоки;
- карточки с мягкими тенями и скруглениями;
- цветовую кодировку статусов: emerald для нормы, amber для внимания, red для высокого риска, blue для справочной информации;
- текущие иконки или их нативные аналоги из `lucide-react-native`;
- структуру карточек, таблиц, бейджей, прогресс-баров и ring-индикаторов.

### 5.2. Правило UI parity

Перед переписыванием каждый экран из `project` должен быть зафиксирован как эталон:

- скриншот исходного экрана;
- список блоков сверху вниз;
- состояния загрузки, ошибки, пустого результата и готового результата;
- размеры отступов, типографика, цвета, состояния кнопок;
- навигационные переходы.

Критерий приемки: новый экран в мобильном приложении визуально совпадает с эталоном на ключевых разрешениях iPhone и Android минимум на уровне ручной QA-проверки и screenshot review.

## 6. Функциональные требования к новой версии

### 6.1. Авторизация и профиль

- Регистрация, вход, выход.
- Восстановление доступа.
- Профиль здоровья: имя, пол, возраст, рост, вес, группа крови, хронические заболевания, аллергии.
- Расчет ИМТ на клиенте и хранение исходных данных в базе.

### 6.2. Анализы

- Загрузка PDF, JPG, PNG, HEIC.
- Съемка камерой.
- Статусы обработки: `uploaded`, `ocr_processing`, `ai_processing`, `completed`, `failed`.
- Список извлеченных показателей с нормами, единицами, статусами и рекомендациями.
- История анализов.
- Динамика отдельных показателей.

### 6.3. AI и OCR

- OCR должен выполняться на backend-стороне или через защищенного провайдера.
- AI-запросы выполняются только через Edge Functions.
- Результаты AI должны сохраняться как структурированные данные, а не только текст.
- Должна быть возможность отключить AI/OCR через admin settings.
- Нужны лимиты по тарифам и защита от повторной обработки одного файла.

### 6.4. Риски, лекарства, программы

- Оценка рисков на основе профиля и анализов.
- Расписание лекарств с push-напоминаниями.
- Программы здоровья с прогрессом, дневными заданиями, питанием и рекомендациями.
- Справочник показателей с поиском и категориями.

### 6.5. Отчет для врача

- Генерация PDF на backend.
- Share sheet в мобильном приложении.
- Временная ссылка для врача с ограничением доступа.
- Отдельная пометка, что отчет не является диагнозом.

### 6.6. Администрирование

Админ-панель не должна оставаться частью обычного мобильного пользовательского интерфейса. Рекомендуется отдельный web-admin:

- управление AI/OCR-провайдерами;
- модели, лимиты, температура, max tokens;
- включение/отключение функций;
- просмотр статистики;
- аудит изменений;
- управление программами здоровья и справочником.

## 9. Риски и открытые вопросы

- Нужно уточнить юридические требования по хранению медицинских данных в целевых странах.
- Нужно выбрать конкретных OCR и AI-провайдеров с учетом стоимости, качества русского языка и медицинских документов.
- Нужно решить, будет ли сервис медицинским информационным продуктом или попадет под более строгую регуляцию.
- Нужно определить тарифную модель до реализации платежей.
- Нужно решить, нужен ли web-кабинет врача или только share-ссылка на отчет.

## 10. Итоговая рекомендация

Переписывать текущий проект следует не как перенос Vite-сайта в WebView, а как полноценное React Native/Expo-приложение. Такой подход сохраняет скорость разработки благодаря React/TypeScript, но дает нативный доступ к камере, файлам, push-уведомлениям, биометрии, share sheet, публикации в App Store и Google Play.

`project` должен остаться эталонной папкой: из него берутся экраны, визуальные решения, демо-данные, доменная модель и SQL-миграции. Новая реализация должна строиться рядом, в отдельной структуре, с production backend-слоем на Supabase Edge Functions и строгой защитой медицинских данных.
